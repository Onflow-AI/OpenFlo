Beyond Leaderboards: Why Current SRE Benchmarks Fail to Measure What Production Actually Demands

Joe|April 2026

Position Paper

Every existing benchmark for evaluating language models on SRE tasks measures knowledge about reliability engineering. None measure execution of it under realistic production conditions.

What Benchmarks Currently Test

The landscape of SRE-oriented LLM evaluation is nascent but growing. The most substantive work to date is Rootly AI Labs' SRE-skills-bench, which evaluates models on multiple-choice infrastructure questions (Terraform, Kubernetes, IAM policy) across AWS, GCP, and Azure, and on the GMCQ task, matching a bug description to the correct pull request diff.

But it shares a structural limitation with every benchmark in this space: tasks are static, isolated, and answer-keyed. A model reads a question and selects a response. There is no broken system. There are no cascading effects. There is no cost to a wrong action taken mid-investigation.

Level 1

Knowledge (MCQ)

Models answer questions about infrastructure concepts. Fast to run, easy to score, but measures recall rather than reasoning under uncertainty.

Level 2

Static Agentic Tasks

Models are given a broken configuration, log dump, or code diff and must produce a diagnosis or fix. Closer to real work, but the system state is frozen — the model cannot take actions, observe effects, or course-correct.

Level 3

Live Environment Simulation

Models operate inside a running, instrumented system. They call real tools, observe real telemetry, and the environment responds to their actions. This level does not yet exist as a standardised benchmark.

Why Levels 1 and 2 Are Not Enough

Production SRE is fundamentally a sequential decision problem under uncertainty. The failure mode of an AI SRE is not answering a Terraform MCQ incorrectly — it is misidentifying a root cause under time pressure, applying a fix that masks the symptom while worsening the underlying fault, or failing to recognise when it lacks sufficient signal to act.

Three properties of real incidents are structurally absent from current benchmarks:

1

Causality Depth

Real distributed failures are multi-hop. A slow database query causes retry storms, which cause thread pool exhaustion, which causes a frontend timeout. Correctly attributing blame requires the model to reason across service boundaries and time, not simply match a symptom to a known pattern.

2

Tool-Mediated Observation

In production, a model does not receive a curated log dump. It must decide which logs to query, which metrics to pull, and in what order. The information-gathering strategy is itself part of the evaluation.

3

Action Consequence

Restarting a pod, rolling back a deployment, or scaling a service changes the system state. A benchmark that cannot model these effects cannot measure whether a model reasons safely about blast radius — one of the most critical properties for production autonomy.

The Level 3 Vision

A Level 3 SRE benchmark requires four components working together.

📦

Reproducible Environment

A fixed, containerised distributed system topology — frontend, API tier, database, message queue, sidecar proxy — all instrumented with OpenTelemetry traces, Prometheus metrics, and structured logs.

⚡

Fault Library

Programmatic fault injection spanning resource exhaustion, network failures, configuration errors, and emergent cascade failures. Each fault is tagged with its ground-truth causal chain.

🛠️

Tool Interface

A constrained, auditable tool interface: log queries, metric lookups, kubectl commands, service restarts. Every tool call is logged, creating a full action trace for evaluation.

📊

Evaluation Function

Multidimensional scoring across outcome, RCA accuracy, step efficiency, safety, and epistemic calibration.

Scoring Dimensions

  • Outcome — Did the system return to a healthy state within N turns?
  • RCA Accuracy — Did the model's stated root cause match the injected fault?
  • Step Efficiency — How many tool calls were required?
  • Safety — Did the model take any actions that worsened system state?
  • Epistemic Calibration — Did the model correctly identify when it lacked sufficient signal, or did it act on incomplete information?

The last metric is arguably the most important for production deployment. A model that confidently misdiagnoses is more dangerous than one that says "I need more data."

Open Research Questions

A well-designed Level 3 benchmark opens empirical questions that have no current answers:

  • Does chain-of-thought reasoning improve SRE performance more than it improves coding performance?
  • Do models degrade gracefully as causal chain length increases from one hop to five?
  • Can models transfer reasoning from one cloud provider's failure vocabulary to another's?

These questions have direct product implications for any team building AI agents on production infrastructure.

A Call to Action

SWE-bench succeeded because it had a clear evaluation contract, a reproducible harness, and open data. The same is achievable for SRE — but it requires treating incident response as an environment problem, not a question-answering problem.

The benchmark that does this first will set the standard for how the industry evaluates AI reliability agents for the next several years.
